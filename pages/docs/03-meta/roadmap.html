--- title: InstantClick’s roadmap ---

Roadmap

As you may have noticed, InstantClick’s development has been stagnating HARD in the last six months. It’s because the docs needed an overhaul, and I thought I needed to tackle all the design decisions for the next version of InstantClick.

The new docs are there. For design decisions, I’ll tackle them one by one and not all at once. The project should get moving again.

Bugs

InstantClick 3.1 should be focused on bug fixes. There are quite a few of them, and those that break normal browser behaviors aren’t acceptable.

There are also surely other bugs that haven’t been reported in InstantClick yet. Wading through pjax’s and Turbolinks’s issues should uncover them. That’s maybe not for 3.1 though.

Miscellaneous

  • The most needed feature is to allow preloading on links added by JavaScript. #41
  • Add more events.
  • Decoupling the progress bar from InstantClick’s core, so that there isn’t useless code lying around for those that wish to use another progress indicator. It will also insure that the included progress bar doesn’t get unfair advantages over custom loading indicator, as is the case now.
  • Make the progress bar appear only after the user sees a page loading for 80–100 ms. #58
  • Make the progress bar show real progress, if the Content-Length header is available. And tell how to activate this header with PHP.
    • Send a header with InstantClick’s requests. #71
      This would bring very interesting possibilities, such as inlining CSS in the HTML so that serving your website takes only one HTTP request.
    • Allow using any element as the container, so you can send just that part to requests made with InstantClick and thus avoid wasting server resources.
    • Make it work with forms (pjax, not preloading). For now it’s not a priority, but it will be needed for that previous thing about inlining CSS. #74
  • Remove whitelist mode and make a better assets change tracking system. Those two got bad design decisions. Probably for 4.0.
  • A page of documentation about caching. Caching is the most requested feature in InstantClick, yet browsers already have all that is needed for caching, and it works great with InstantClick.

On the project

  • A list of fixes for third-party scripts with InstantClick, much like Turbolinks Compatibility, is badly needed.
  • An Expo/showcase of sites using InstantClick would be great, because it’s not really if and how much InstantClick is used in the wild; even I don’t track that much. You know what? If you’d like to be in the expo, start sending your site’s URL now to adieulot à gmail , com.
  • As a maintainer of an OSS project, I have no idea what I’m doing. I need to progressively get better at it (POSS is a good resource). The most pressing thing is to learn how to coordinate different development efforts at the same time, for now it’s been pretty much just me developing in the master branch.
  • A forum for getting help with InstantClick.

Nothing on this page is set in stone, if you have feature or design requests you’re invited to discuss them on GitHub.
